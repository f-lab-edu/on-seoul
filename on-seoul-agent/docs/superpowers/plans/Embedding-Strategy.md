# 임베딩 개선 전략

## 개요

> on-seoul 하이브리드 검색에서 **임베딩 품질**을 끌어올리기 위한 전략 문서. 무엇을 임베딩할지(청킹, 추출), 어떻게 검색할지(쿼리 변환)를 데이터 특성에 맞게 설계한다.

기존 임베딩(`{area_name} {category} {service_name}`)이 시설 식별형 질의에서는 잘 동작하지만, 의미/맥락형 질의와 세부정보형 질의에서 한계를 보이는 문제를 해결한다.

---

## 문제 정의

### 데이터 특성

서울시 공공서비스 예약 데이터를 분석한 결과 두 가지 두드러진 특성이 보인다.

**시설 메타데이터는 짧고 정형화되어 있다.** `service_name`, `area_name`, `max_class_name`, `min_class_name`은 수십 단어 수준이며 패턴이 정해져 있다. 시설 식별을 위한 임베딩에는 청킹이 불필요하고, 현재 단순 concat 방식으로도 충분히 동작한다.

**반면 `detail_content`는 매우 길고 노이즈가 많다.** 평균 2,000~5,000자, 일부는 10,000자 이상이며, 다음과 같은 동일한 설명 문장이 60~70%를 차지한다.

- "1. 공공시설 예약서비스 이용시 필수 준수사항…"
- "본 사이트와 각 관리기관의 규정을 위반할시에는…"
- 약관/면책조항/일반 안내 등

실제 검색에 유용한 정보(이용료, 운영시간, 부대시설, 대상, 취소 규정)는 본문 중간에 짧게 흩어져 있다.

### 질의 패턴

임의로 생성한 평가셋 100개 질의를 분석하면 검색 전략이 다른 세 부류로 나뉜다.

| 부류 | 비율 | 예시 | 현재 한계 |
|---|---|---|---|
| 시설 식별형 | 약 60% | "마포구 풋살장", "응봉공원 테니스장" | 큰 문제 없음 |
| 의미/맥락형 | 약 30% | "아이랑 갈 수 있는 무료 체험", "드론 날릴 수 있는 곳" | 사용자 의도와 문서 어휘 격차로 Recall 저하 |
| 세부정보형 | 약 10% | "테니스장 평일 이용료", "취소는 며칠 전까지" | `detail_content`가 검색 가능한 형태로 색인되어 있지 않음 |

## 전략 1: 청킹 대신 문서 증강

### 결론
시설 메타데이터에는 **청킹을 적용하지 않는다.** 대신 **문서 증강(Augmentation)** 으로 검색 가능성을 높인다. 모든 처리에 앞서 **변별성 없는 텍스트를 사전에 제거**하고, 다음 세 가지 기법을 병행한다.

- **사전 정제 (Pre-filtering)**: `detail_content`에서 모든 시설에 공통된 boilerplate를 마커 기반으로 제거. 후속 LLM 추출, 임베딩, BM25 색인의 입력 품질을 결정한다.
- **Metadata Enrichment**: 정제된 텍스트에서 LLM으로 핵심 항목을 추출해 `metadata.extracted`에 저장. BM25 색인 대상이 된다.
- **Summary/Question Augmentation**: 추출된 한 줄 요약(`extracted.summary`)과 LLM이 생성한 예상 질문을 별도 임베딩해 의미 매칭 보조 인덱스로 활용한다.

### 왜 청킹을 하지 않는가

전통적인 슬라이딩 윈도우 청킹(256/512 토큰)을 `detail_content`에 적용하면 다음 문제가 발생한다.

| 문제 | 영향 |
|---|---|
| 공통 설명 문장이 청크 대부분을 차지 | 모든 시설 청크가 비슷해져 변별력 하락 |
| 한 시설이 N개 청크로 분리 | 결과 페이지에 같은 시설 중복 노출 |
| 청크 간 컨텍스트 손실 | "야주개홀 220석"이 분리되면 매칭 실패 |
| 저장, 검색 비용 증가 | 임베딩 N배, HNSW 인덱스 크기 N배 |

**시멘틱 청킹도 적합하지 않다.** 슬라이딩 윈도우의 대안인 시멘틱 청킹(문장 임베딩 유사도로 경계를 찾는 방식)도 이 데이터에서는 효과가 제한적이다.

- `detail_content`의 60~70%가 모든 시설에 공통된 동일한 설명 문장이라, 문장 간 의미 거리가 작아 경계 탐지가 부정확하다.
- 시설 메타데이터는 항목 단위로 의미가 명확히 구분되어 있어(이용료, 운영시간, 부대시설 등), 비결정적인 시멘틱 분할보다 LLM이 항목을 직접 추출하는 쪽이 결과가 안정적이다.
- 시멘틱 청킹은 청킹 자체의 구조적 단점(중복 노출, 컨텍스트 손실, 저장 비용)을 해결하지 못한다.

청크의 장점(세부 정보 검색)은 살리되 단점은 피하기 위해 **구조화 추출**을 사용한다.

### 사전 처리: 불필요 텍스트 소거

#### 사전 처리가 선행되어야 하는 이유

**데이터 내용의 특성:**`detail_content`의 60~70%는 모든 시설에 공통된 안내문이다. 임의로 10개 시설을 추출해 비교하면 다음 텍스트가 반복적으로 등장한다.

| 반복 출현 텍스트 | 출현 비율 |
|---|---|
| "1. 공공시설 예약서비스 이용시 필수 준수사항…" 단락 | ~100% |
| "2. 시설예약 비회원일 경우…" 단락 | ~100% |
| "4. 주의사항" 도입부의 공통 안내 | 체육시설 다수 |
| "영업행위, 양도, 양수, 교환, 판매행위를 금지" 류 | 70%+ |
| "5초이내 예약건에 대해서는 비정상예약으로…" | 60%+ |

이를 그대로 임베딩하거나 LLM 추출에 입력하면 다음 문제가 발생한다.

- 임베딩 벡터의 변별력이 사라진다.
- BM25에서 "취소", "예약", "이용" 같은 핵심어의 IDF가 0에 수렴해 기여도가 0이 된다.
- LLM 입력시 토큰 사용량과 처리시간은 늘어나지만, 정작 변별력이 필요한 정보는 누락되거나 짧게 추출될 수 있다.

따라서 사전 정제는 어떤 임베딩/추출 전략보다도 먼저 적용한다.

#### 구현 방식 - 마커 기반 구간 추출

`detail_content`는 일관된 구조를 따른다(1. 공공시설 준수사항 → 2. 시설예약 안내 → 3. 상세내용 → 4. 주의사항). 변별 정보는 거의 전부 **3번 섹션**에 모여 있으므로 이 구간만 추출한다.


```python
# 예시 코드
start = text.find("3. 상세내용")
end = text.find("4. 주의사항")
if start >= 0:
    cleaned = text[start + len("3. 상세내용"):end if end > start else None]
```

- "3. 상세내용" 이후만 사용해 1번/2번 boilerplate(가장 큰 덩어리) 제거.
- "4. 주의사항" 이전 컷으로 잔존 boilerplate의 60%+ 추가 제거.

환불 비율표는 형식이 표준화되어 있어 임베딩에서 노이즈로 작용하지만, **별도 structured 필드는 만들지 않는다.** 평가셋에서 환불 관련 질의의 recall이 부족한 것이 확인되면 그때 도입한다.

#### 적용 범위

| 단계 | 사전 정제 적용 |
|---|---|
| Track A 식별 임베딩 입력 | 불필요 (이미 변별 필드만 사용) |
| Track B summary 추출 LLM 프롬프트 | **필수** |
| Track C 예상질문 생성 LLM 프롬프트 | **필수** |
| BM25 색인 대상 `metadata.extracted` | **필수** (추출 결과에 boilerplate 잔존 시 IDF 오염) |

### 구조화 추출 설계

임베딩 적재 배치(`embed_metadata.py`)에 LLM 호출을 추가해 **사전 정제된 텍스트**에서 핵심 항목을 추출하고 `metadata` JSONB에 저장한다.

**추출 항목**
- `fee`: 이용료 (평일/주말 구분 포함)
- `operating_hours`: 운영시간
- `cancellation`: 취소 규정 (며칠 전까지 가능, 환불 비율)
- `facilities`: 부대시설 (빔프로젝터, 마이크 등)
- `capacity`: 수용 인원
- `restrictions`: 이용 제한 (금연, 음식물 반입 등)
- `summary`: 시설 한 줄 요약 (다음 절에서 사용)

**저장 위치**
`service_embeddings.metadata` JSONB 안에 `extracted` 키로 저장한다. 새 컬럼이나 테이블은 추가하지 않는다. BM25 인덱스가 이미 `metadata`를 JSON 필드로 색인하므로, 추출된 정보가 자동으로 BM25 검색 대상이 된다.

### 삼중 트랙 임베딩

추출된 정보를 최대한 활용하기 위해 세 종류의 임베딩을 운영한다.

| 트랙 | 입력 텍스트 | 용도 |
|---|---|---|
| **Track A: 식별 임베딩** (현행 개선) | `{area_name} {max_class_name} {min_class_name} {service_name} {place_name}` | 시설 식별형 질의 |
| **Track B: 요약 임베딩** | `extracted.summary` (사전 정제된 detail_content에서 LLM 추출) | 세부정보형 및 일부 의미/맥락형 질의 |
| **Track C: 예상질문 임베딩 (HyQE)** (신규) | LLM이 row마다 생성한 5~15개의 예상 질문 텍스트 | 의미/맥락형 질의 |

`service_embeddings`를 row-per-vector 통합 테이블로 재정의한다. 별도 컬럼/테이블을 만들지 않고, 각 트랙이 `row_kind` 디스크리미네이터로 같은 테이블 안에 들어간다.

- Track A → `row_kind='identity'` (시설당 1행)
- Track B → `row_kind='summary'` (시설당 0~1행, LLM 실패 시 미생성)
- Track C → `row_kind='question'` (시설당 0~N행, HyQE 결과)

모든 row가 동일한 `embedding`(vector(768)) 컬럼과 `embedding_text`(실제 임베딩 텍스트) 컬럼을 갖는다. Track C는 한 시설에 다수 row가 존재하므로 검색 SQL의 `ROW_NUMBER() OVER (PARTITION BY service_id)` 로 dedup한다. 구체 DDL은 [구현 계획서](./2026-05-18-triple-track-embedding-pipeline.md) 및 `scripts/ddl/service_embeddings.sql` 참고.

#### 왜 Track C(HyQE)가 필요한가

Track A는 시설명/지역/분류에 등장하는 키워드만 잡는다. Track B의 요약은 변별 정보의 압축이라 안정적이지만 본질적으로 정보를 잃는다 — "반딧불이 체험"이 `detail_content` 한 줄에 있어도 요약에서 빠질 수 있다. 의미/맥락형 질의(드론, 전통문화 등)는 이 두 트랙 모두에서 누락 가능성이 높다.

Track C는 다음 메커니즘으로 이 공백을 메운다.

- **질문 ↔ 질문 매칭이 질문 ↔ 문서 매칭보다 벡터 공간 분포가 정합한다.** 사용자 쿼리와 인덱스가 같은 분포(질문 형식)에 있으면 코사인 유사도가 안정적으로 높아진다.
- **변별성이 자동 보장된다.** LLM에 "이 시설에 대한 사용자가 할 만한 질문"을 시키면 자연스럽게 그 시설만의 특징(반딧불이, 짚공예 등)에 집중한다.
- **청킹의 단점이 발생하지 않는다.** 한 시설이 N개 임베딩으로 분리되지만 `service_id` 중복제거로 해결되며, 컨텍스트 손실 문제는 애초에 없다(질문은 self-contained).

#### 질문 생성 분포

100% 의미/맥락형으로 쏠리지 않도록 HyQE 생성 시 의도 분포를 명시한다.

| 의도 유형 | 비율 | 예시 |
|---|---|---|
| 의미/맥락형 | 50% | "{활동} 어디서 해?", "{대상}이 갈 만한 {경험}" |
| 세부정보형 | 30% | "{시설} 이용료", "취소 며칠 전까지" |
| 변형/짧은 질의 | 20% | "드론", "반딧불이", "짚공예" (키워드형) |

이 비율은 평가셋 100개의 의도 분포와 정합한다. 식별형 비중을 의도적으로 낮춘 것은 Track A와 BM25가 이미 해당 영역을 커버하기 때문이다.

#### 평가셋의 분리 활용

평가용 100개 질의를 다음과 같이 분리한다.

- **80개는 평가셋으로 봉인** — recall@k, MRR 측정 전용. 학습/프롬프트에 노출하지 않는다.
- **20개는 HyQE 생성 프롬프트의 few-shot 라벨**로 활용. 의도유형 컬럼을 함께 넣어 분포를 강제한다.

전체를 인덱스에 직접 넣으면 평가셋에 overfit된 시스템이 된다.

### FAQ, 가이드 문서는 별도 인프라

가이드성 질의("예약 변경 어떻게 해?", "결제 안 돼")는 시설 메타데이터에 없는 정보이며, 별도로 등록·관리되는 FAQ(현재 6건)로 답해야 한다.

URL: [서울 공공서비스 예약](https://yeyak.seoul.go.kr/web/bbs/selectPageListBbs.do?bbs_code=A1001)


**청킹은 적용하지 않는다.** 실제 FAQ는 한 건이 한 주제만 다루고, 평균 100~500자로 짧으며, 단순 Q/A 평문이다. 청킹이 해결해야 할 문제(여러 주제 혼재, 컨텍스트 손실)가 존재하지 않으므로 Q/A 전체를 하나의 단위로 임베딩한다.

**검색은 Router Agent에서 분기한다.** 질의 의도가 `가이드 문의`형이면 시설 검색을 거치지 않고 FAQ 검색만 수행한다. 시설명이 함께 언급된 경우는 시설 검색으로 보내 `metadata.extracted.cancellation` 등을 활용한다.

FAQ 검색 자체는 단순 벡터 검색으로 충분하다. 6건 규모에서 BM25 인덱스나 HyDE는 과한 투자이며, 향후 수십 건 이상으로 늘어나면 재검토한다.

## 전략 2: HyDE 기반 쿼리 변환

### 결론
> 우선 전략 1만 구현하고 모니터링 후 필요할 경우 도입한다.
> 적용하더라도 **의미/맥락형 질의에 한하여** HyDE(Hypothetical Document Embeddings)를 적용한다. 고유명사형, 정형 필터형 질의에는 적용하지 않는다.

### HyDE 도입 배경

이 데이터셋에서 사용자 질의와 문서 사이의 **어휘 격차**가 크다.

| 사용자 질의 | 실제 문서 |
|---|---|
| "아이랑 갈 수 있는 무료 체험" | `target_info: 어린이(단체)`, `payment_type: 무료` |
| "한강에서 영화 촬영하려면?" | "영화촬영(주간)-광나루한강공원", "CF, 뮤직비디오 촬영시 예약" |
| "드론 날릴 수 있는 곳" | "광나루 한강 드론장(회전익 드론 구역)" |

사용자는 **의도/동사/자연어**로 표현하고, 문서는 **공식 명칭/명사/행정용어**로 작성되어 있다. 원문 질의를 그대로 임베딩하면 벡터 공간에서 거리가 멀어진다.

HyDE는 LLM이 질의를 **문서 어휘로 재작성한 가상 문서**를 만들고, 그 가상 문서를 임베딩해 검색에 사용한다. 예시로 "아이랑 갈 수 있는 무료 체험"이 "마포구 어린이 가족 대상 무료 문화체험 교육 프로그램"으로 변환되어 실제 문서와 가까워진다.

이 데이터셋은 시설 메타데이터의 형식이 정해져 있어 LLM이 가상 문서를 잘 흉내 낼 수 있다는 점에서 HyDE 적용에 유리하다.

### 전략 1의 Track C(HyQE)와의 관계

> HyDE와 HyQE는 어휘 격차를 좁힌다는 목표는 같지만 방향이 반대다.

- **HyDE**: 질의 측을 문서 어휘로 변환(질의 → 가상 문서).
- **HyQE**: 문서 측을 질의 어휘로 변환(문서 → 가상 질의).

두 기법은 충돌하지 않고 상호 보완적이다. HyQE 임베딩은 색인 시점에 미리 생성되므로 검색 latency에 영향을 주지 않는다. HyDE는 검색 시점의 추가 변환이라 보완적인 recall을 제공한다.

### 질의 유형별 적용 여부

| 적용 | 유형 | 예시 |
|---|---|---|
| ✅ | 의미/맥락형 | "아이랑 갈 수 있는 무료 체험", "드론 날릴 수 있는 곳" |
| ✅ | 복합형 | "강남 근처 무료 문화행사", "주말에 가족이랑 갈 만한 박물관" |
| ✅ | 유사도 탐색 | "접수종료됐는데 비슷한 프로그램 있어?" |
| ❌ | 고유명사형 | "서울역사박물관 야주개홀", "따릉이 대여소" |
| ❌ | 정형 필터형 | "접수 중인 공간시설", "5월 31일까지 접수 가능" |
| ❌ | 세부정보형 | "테니스장 평일 이용료", "청계천박물관 전화번호" |

고유명사형에 HyDE를 적용하면 LLM이 원문에 없는 단어를 생성해 오히려 노이즈가 된다. 정형 필터형, 세부정보형은 SQL이나 BM25로 처리되므로 HyDE의 이득이 없다.

**중요**: HyDE는 **벡터 검색에만** 적용한다. BM25는 정확 토큰 매칭이 강점이므로 원문 질의를 그대로 사용한다.

### 프롬프트 설계

일반 HyDE 프롬프트("이 질문에 답하는 문서를 작성하세요")는 너무 자유로워서 노이즈가 크다. **도메인 제약**을 거는 것이 핵심이다.

Few-shot, 시드 고정등을 활용하여 가상 문서의 형식은 **임베딩 텍스트 구성과 동일하게** 맞춘다.

```
형식: {지역구} {대분류} {소분류} {시설명} 대상:{대상} 요금:{유무료}
```

이렇게 하면 가상 문서와 실제 임베딩 텍스트의 어휘, 구조가 정렬되어 코사인 유사도를 향상시킬 수 있다.

### 의도 분류와의 통합

기존 `VectorAgent`의 `_RefinedQuery` 추출 단계에서 **HyDE 가상 문서까지 함께 생성**한다. 별도 LLM 호출 없이 한 번의 호출로 처리한다.

```
LLM 입력: 사용자 질의
LLM 출력: {
  refined_query: "...",
  intent_type: "semantic",
  hyde_document: "마포구 교육강좌 자연/과학 유아숲체험원 어린이 단체 대상 무료",
  filters: { area_name, max_class_name, ... }
}
```

`intent_type`이 `semantic`, `complex`인 경우에만 `hyde_document`를 임베딩 입력으로 사용한다. 그 외에는 `refined_query`를 직접 임베딩한다.

### Multi-HyDE를 채택하지 않는 이유

가상 문서를 N개 생성해 임베딩 평균을 내는 Multi-HyDE는 비용이 N배지만 효과는 제한적이라는 보고가 있다. 단일 HyDE로 시작해 모니터링 및 프롬프트 개선 작업 후에도 한계가 명확해진 뒤에 도입여부를 검토한다.

### HyDE 대안: Query Expansion

HyDE가 부담스럽거나 응답 시간 제약이 있는 경우 **Query Expansion**이 가벼운 대안이다.

| 항목 | HyDE | Query Expansion |
|---|---|---|
| LLM 의존 | 필요 | 선택 (사전 기반 가능) |
| 응답 시간 | +200~500ms | +0~100ms |
| 효과 | 높음 | 중간 |
| 디버깅 | 어려움 (비결정적 생성) | 쉬움 (사전 명시적) |

처음에는 Query Expansion으로 시작해 한계를 확인한 뒤 HyDE로 넘어가는 점진적 도입도 가능하다.

## 전체 임베딩 파이프라인

두 전략을 통합한 최종 흐름은 다음과 같다.

### 색인(Indexing) 단계

```
시설 메타데이터
    │
    ├─→ Track A 텍스트 구성 → Gemini Embedding → service_embeddings (row_kind='identity', 1행)
    │   ({area_name} {max_class_name} {min_class_name} {service_name} {place_name})
    │
    └─→ detail_content
            │
            └─→ 사전 정제 ("3. 상세내용" ~ "4. 주의사항" 구간 추출)
                    │
                    ├─→ LLM 구조화 추출
                    │       ├─→ identity row의 metadata.extracted (JSONB, BM25 색인 대상)
                    │       └─→ extracted.summary
                    │
                    ├─→ Track B 텍스트 = extracted.summary → Gemini Embedding → service_embeddings (row_kind='summary', 0~1행)
                    │
                    └─→ Track C 텍스트 = LLM 생성 예상 질문 N개 → Gemini Embedding → service_embeddings (row_kind='question', 0~N행)
```

> 세 트랙 모두 동일한 `service_embeddings` 통합 테이블에 row-per-vector 로 적재된다. 별도 테이블/컬럼 없음.

### 검색(Retrieval) 단계

```
사용자 질의
    │
    ├─→ LLM 의도 분류 + 정제 + HyDE 생성 (단일 호출)
    │       ├─→ intent_type
    │       ├─→ refined_query
    │       ├─→ filters
    │       └─→ hyde_document ([Phase 2] - 결과 모니터링 이후 semantic/complex에서만)
    │
    ├─→ intent_type에 따라 분기
    │       ├─→ guide_inquiry → FAQ 벡터 검색 (별도 인프라)
    │       ├─→ semantic/complex
    │       │       ├─→ embed(refined_query) → 시설 검색      [Phase 1]
    │       │       └─→ embed(hyde_document) → 시설 검색      [Phase 2]
    │       └─→ 그 외 → embed(refined_query) → 시설 검색
    │
    ├─→ pgvector 검색 (service_embeddings 통합 테이블에서 row_kind 별 partial 쿼리)
    │       ├─→ WHERE row_kind='identity' (Track A): 시설 식별
    │       ├─→ WHERE row_kind='summary'  (Track B): 의미 매칭 보조
    │       └─→ WHERE row_kind='question' (Track C, PARTITION BY service_id): 의미/맥락형 매칭
    │
    ├─→ BM25 검색 (원문 질의 사용, HyDE 미적용, partial index — WHERE row_kind='identity')
    │       └─→ service_name + metadata (구조화 추출 정보 포함)
    │
    └─→ RRF 결합 (service_id 중복 제거) → 최종 순위
```


### RRF 결합 전략

> 세 트랙의 벡터 검색 결과와 BM25 결과를 RRF로 결합한다. 가중치 결정 방식, Router Agent 확장(`VectorSubIntent` 분류 추가), 의도별 가중치 프로파일 등 결합 단계의 상세 설계는 별도 문서에서 다룬다.

별도 문서 참고 → [RRF 결합 전략](./RRF-Strategy.md)

## 기대 효과

> 각 전략별로 어느 질의 유형에 효과가 있는지 예상한다.

| 전략 | 시설 식별형 | 의미/맥락형 | 세부정보형 |
|---|---|---|---|
| 사전 정제 (변별성 소거) | - | 보조 | 보조 |
| Track A 임베딩 개선 (place_name 추가) | 소폭 개선 | - | - |
| Track B 요약 임베딩 | - | 개선 | 개선 |
| Track C 예상질문 임베딩 (HyQE) | - | **큰 개선** | 개선 |
| 구조화 추출 → BM25 | - | - | **큰 개선** |
| [Phase 2] HyDE | - | **큰 개선** | - |

전체적으로 **의미/맥락형과 세부정보형에서 큰 개선**이 기대된다. 시설 식별형은 이미 잘 동작하므로 회귀가 없는 것이 중요하다.

## 리스크와 한계

| 리스크 | 영향 | 대응 |
|---|---|---|
| LLM 추출 품질 편차 | 일부 시설의 `summary`가 부정확 | 추출 결과 샘플 검수, 프롬프트 반복 개선 |
| 사전 정제 마커 누락 | 일부 row에 "3. 상세내용" dnt마커 부재 시 raw text가 흘러감 | fallback 처리(마커 없으면 전체 사용), 누락률 모니터링 |
| Track C 질문 생성 편향 | 의미/맥락형에 쏠리면 식별/세부정보 recall 회귀 | 의도 분포 강제(50/30/20), few-shot 라벨 활용 |
| 추출 비용 (Gemini API) | 전체 시설 × 1회 호출 (Track B + C 합산) | 배치 처리, 변경된 시설만 재추출 |
| [Phase 2] HyDE 생성 시간 (+200~500ms) | 응답 지연 | 의도 분류와 통합해 추가 호출 0회로 설계 |
| 고유명사형에 HyDE 오적용 | Recall 회귀 | 의도 분류 정확도 검증, 평가셋으로 회귀 측정 |
| 세 임베딩 컬럼 → 저장 공간 증가 | 디스크/메모리 비용 | 효과 검증 후 도입. Track C는 row 분리로 추가 부담이 큼 |
| Track C 결과 중복 노출 | 같은 시설이 결과 페이지에 N번 등장 | `service_id` 기준 중복 제거, 최고 rank만 유지 |

## 결정 사항 요약

1. 시설 메타데이터에는 **청킹을 적용하지 않는다.** (슬라이딩 윈도우, 시멘틱 청킹 모두 부적합)
2. `detail_content`는 **마커 기반 사전 정제** 후 LLM 처리에 입력한다. ("3. 상세내용" ~ "4. 주의사항" 구간 추출)
3. `detail_content`는 **LLM 구조화 추출**로 처리해 `metadata.extracted`에 저장한다.
4. 임베딩은 **삼중 트랙**으로 운영한다. Track A(식별)는 현행 개선, Track B(요약)는 추출된 summary, Track C(HyQE)는 LLM 생성 예상 질문을 사용한다.
5. **HyDE는 바로 도입하지 않는다.** 전략 1 구현 및 모니터링 후 필요할 경우 **의미/맥락형 질의에 한하여** 조건부 적용한다. 의도 분류와 통합해 추가 LLM 호출 없이 처리한다.
6. **BM25는 원문 질의를 사용**한다. HyDE는 벡터 검색 전용이다.
7. **FAQ는 별도 인프라로 분리**하고, 청킹 없이 Q/A 단위로 임베딩한다. Router Agent가 시설 검색과 분기한다.
8. **RRF 결합 방식 및 가중치 프로파일은 [별도 문서](./RRF-Strategy.md)** 에서 다룬다. 측정 기반 결정, Router Agent 확장(`VectorSubIntent` 추가), 의도별 가중치 적용을 포함한다.
9. **평가셋 100개 중 80개는 봉인**한다. 20개만 HyQE 생성 프롬프트의 few-shot으로 활용한다.
